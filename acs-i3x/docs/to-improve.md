# i3X Service — To-Improve Document (TID)

Tracked deviations from the i3X spec and known areas for improvement.

## Spec Deviations

| ID | Area | Deviation                                                | Reason                                                                                    | Resolution path                                              | Status |
|----|------|----------------------------------------------------------|-------------------------------------------------------------------------------------------|--------------------------------------------------------------|--------|
| D1 | Auth | Using HTTP Basic Auth instead of API keys (MUST in spec) | PoC simplicity — Basic Auth maps directly to Factory+ username/password via ServiceClient | Implement API key / Bearer token auth with principal mapping | Open   |
| D2 | Response Format | Error envelope omits `error.code`. Spec requires `{ success: false, error: { code, message } }`; implementation returns only `{ message }` (`lib/middleware/envelope.ts:39`) | HTTP status is set correctly but never mirrored into the body. Overlooked when the error handler was first written | Add `code: status` alongside `message` in the error JSON body | Resolved |
| D3 | Subscriptions | `POST /subscriptions/delete` returns `{ deleted: [...] }` (`api-v1.ts:427`). Spec requires bulk-results envelope: `{ success, results: [{ success, subscriptionId, result: null }] }` | PoC shortcut — flat array felt simpler than per-id results | Build per-id results in the handler, returning success/error per subscriptionId | Open |
| D4 | Subscriptions | `POST /subscriptions/register` returns `{ registered: [...] }` (`api-v1.ts:435`) and aborts on first bad elementId. Spec requires bulk-results envelope **and** "Server MUST support partial failures (e.g. bad elementId) and not fail the full request" | PoC shortcut; partial-failure handling not yet implemented | Iterate elementIds, collect per-id success/error, return bulk envelope | Open |
| D5 | Subscriptions | `POST /subscriptions/unregister` returns `{ unregistered: [...] }` (`api-v1.ts:444`); same partial-failure gap as D4 | As D4 | As D4 | Open |
| D6 | Subscriptions | `POST /subscriptions/list` returns a flat array of `{ clientId, subscriptionId, displayName }` (`subscriptions.ts:67–80`); spec requires bulk envelope AND a `monitoredObjects: [{ elementId, maxDepth }]` field per subscription | Registered-element detail isn't currently exposed by `SubscriptionManager` | Track per-element maxDepth in the subscription state, expose it from `list`, wrap response in bulk envelope | Open |
| D7 | Subscriptions | Deleted / non-existent subscription returns HTTP 500. Spec: "Subsequent calls to `/sync` or `/stream` for a deleted or non-existent Subscription MUST return 404 Not Found." `clientId` mismatch should be 403 | `getAndVerify` (`subscriptions.ts:213–218`) throws a plain `Error` with no `.status`, so the error handler defaults to 500 | Throw an error with `.status = 404` for missing subscription and `.status = 403` for clientId mismatch (or use `APIError` from `@amrc-factoryplus/service-api`) | Open |
| D8 | Query / Subscriptions | No HTTP 206 path for partial `maxDepth` results. Spec: "If the server can return a partial result … it MUST return HTTP 206." | No server-imposed depth cap currently exists, so the partial-result path was never wired up. Latent deviation that will activate if a cap is introduced | When/if a depth cap is enforced (or composition truncation occurs), call `res.status(206)` and return the partial tree; document the cap in `/info` if applicable | Open |

## Known Limitations to Improve

| ID | Area                        | Current behaviour                                                                              | Ideal behaviour                                                                                | Notes                                                                         | Status |
|----|-----------------------------|------------------------------------------------------------------------------------------------|------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------|--------|
| L1 | Device discovery            | Re-poll ConfigDB/Directory every 60s to detect new devices                                     | Use Directory's PostgreSQL LISTEN/NOTIFY or MQTT `Last_Changed/*` events for real-time updates | Directory already has the mechanism (sql/v7.sql triggers, mqttcli.js:437-445) | Open   |
| L2 | ObjectTypes without schemas | Return empty `{"type": "object"}` schema for ConfigDB classes missing a ConfigSchema app entry | Either require schemas or synthesise from Sparkplug birth certificate structure                | Most real devices will have schemas; this is an edge case                     | Open   |
| L3 | SSE streaming scope | SSE subscriptions only stream values from devices publishing to UNS (requires UNS ingester + ISA-95 config) | Stream from all devices, including those only on Sparkplug | Could subscribe to `spBv1.0/#` directly, or use the historian's Sparkplug data. Needs design work — Sparkplug uses protobuf + aliases which requires birth certificate tracking | Open   |
| L4 | Device identity | Device elementId uses ConfigDB UUID, sub-objects use Instance_UUID. Ideally these would be consistent | Use Instance_UUID throughout, or provide a mapping endpoint | Too late to change for PoC without breaking the parent chain from ConfigDB | Open   |
